# 报告提取与数据对照表规范

本文件规范了中短篇高数据密度报告（研报/财报/行业分析）的文本提取、窗口化阅读以及核心数据对照表的初始化流程。

## 1. 文本抽取与工具调用

专业研报和财报通常采用多栏排版，且含有大量图表。使用常规提取工具易导致左右栏文字交错、表格行线错位。

### 文本抽取与数据导入规范
为了保证执行效率，PDF 转换为 Markdown 的多模态解析任务与本解说稿生成 Skill 解耦（不建议在 Skill 执行过程中调用耗时的云端 API 转换工具）。

1. **读取已转换的 Markdown**：如果该报告已有预先提取出的 Markdown 文件（例如使用外部 MinerU 客户端、网页端或手动解析的文本），应当优先直接载入并将其复制规范化命名为：
   ```text
   output/<project>/text/_extract.md
   ```
2. **本地提取（Fitz 降级策略）**：对于输入的 PDF 报告，若未进行预先转换，Agent 应当直接使用本地的 `PyMuPDF (fitz)` 库进行快速文本提取，禁止使用任何阻塞的云端 MinerU API。提取后同样落盘为 `_extract.md`。
3. **文本落盘**：提取后的文本规范化保存路径统一为：
   ```text
   output/<project>/text/_extract.md
   ```

## 2. 规模估算与阅读计划

提取文本后，使用 Bash 命令统计字符规模（用 `wc -m` 计算字符数而非 `wc -c` 字节数）：

```bash
EXTRACT="output/<project>/text/_extract.md"
wc -m "$EXTRACT"
```

### 阅读覆盖率要求
根据 `wc -m` 统计的文稿总字数（字符数），阅读需满足以下简明要求：
- **总字数 < 80,000 字**：必须 **100% 全文覆盖读完**，禁止任何遗漏。
- **总字数 >= 80,000 字**：必须精读并覆盖 **80% 以上的核心章节**（如摘要、核心财务数据表、竞争分析及结论等）。

### 窗口读取命令（Bash）
使用 `sed` 按行读取指定的字符窗口，防止单窗输出超载：
```bash
START=1
END=200
sed -n "${START},${END}p" "$EXTRACT" | awk '{print NR+('"$START"'-1) "\t" $0}'
```

## 3. 建立数据对照表（Data Reference Sheet）

对于商业与研报，数据精度是生命线，单位错乱（如“百万”写成“十亿”）或趋势颠倒（如“同比下降”写成“同比增长”）是绝对无法容忍的。

在开始撰写任何初稿前，**必须建立并保存数据对照表**：
```text
output/<project>/text/_data_reference.json
```

### 对照表 JSON 契约

```json
{
  "source_file": "output/<project>/text/_extract.md",
  "source_total_chars": 35000,
  "data_policy": "All numerical claims in the script must trace back to this ledger.",
  "key_metrics": [
    {
      "metric_id": 1,
      "metric_name": "2025年第四季度研发投入",
      "value": "12.5亿元",
      "trend": "同比增长15%",
      "source_location": "第142行，表3-1研发支出明细",
      "context_snippet": "第四季度研发费用达12.5亿元，较去年同期的10.87亿元增长15%。",
      "business_implication": "表明公司在AI技术栈上持续加码，但短期内将压低净利润率"
    }
  ]
}
```

### 建立要求
- **提取范围**：对照表应至少包含 10–20 个支撑整篇报告核心结论的**关键数字、百分比、增速或趋势方向**。
- **禁止凭空写稿**：后续写稿及修订阶段，如果需要引用任何数据，必须且只能引用此对照表中的内容。
- **严防单位与方向错误**：在录入对照表时，大模型必须仔细核对：
  - 货币单位（人民币、美元、港币等）。
  - 数值单位（元、千、百万、亿、%）。
  - 变化方向（同比增长、环比增长、同比降幅收窄、由盈转亏等）。
