# 研报文风与 JSON 契约

本文件定义了将专业报告和财报转化为短视频/口播文稿时的写作标准与最终 JSON 输出结构契约。

## 1. 最高写作原则

- **数据为王，绝对精准**：文稿中引用的每一个数字，必须与 `output/<project>/text/_data_reference.json` 严格吻合，绝不准许夸张或四舍五入。
- **翻译数据，而不是罗列数据**：不要像读表格一样读数据。必须把冰冷的数据翻译为观众能听懂的“代价、机会、生活现状或企业危机”。
  * *反面案例*：“公司主营业务毛利率从 20% 降至 15%。”（无感）
  * *正面案例*：“这意味着公司每卖出 100 块钱的产品，到手利润直接缩水了四分之一。在如今的竞争下，这个利润率几乎踩在了生死线上。”
- **硬核因果链**：研报不讲温情故事，讲的是“底层激励”与“利益博弈”。要解释清楚：为什么会出现这个数字？背后是谁在买单？谁在受益？面临什么政策或市场约束？

## 2. 开头（Hook）
- **前 100 字内切入冲突**：禁止自我介绍、寒暄或冗长的行业背景介绍。
- 开篇第一句必须是一个**反直觉的数据对比**、**迫在眉睫的危机**或**直接关联观众钱包的问题**。
- 前 100 字必须提供清晰的“价值承诺”，让观众知道花 3 分钟听完有什么收益（例如获得避坑指南、行业底层风向）。
- 自然提及报告名称或报告主角。

## 3. 结构与节奏
- **因果驱动**：文稿结构按照“核心异动 -> 背后原因 -> 数据支撑 -> 现实代价 -> 结论与启发”递进。
- **认知推进**：大约每 250 字就要有一次认知升级。可以是一个令人心惊的行业黑幕、一个政策约束的拆解、或者一个超出常理的财务操作，绝对避免连续两段只念财报指标。
- **数字精炼**：单段文稿中出现的数字不宜超过 3 个，避免听觉过载。用最核心的 1-2 个宏观数据打透逻辑即可。

## 4. 禁用表达
严禁出现任何带有“AI味”或“书面总结腔”的词汇：
- "这份报告显示……" / "报告指出……"
- "从上述图表中可以看出……"
- "如图所示……"
- "本节我们来探讨……"
- "总而言之 / 综上所述……"
- "建议转发给有需要的人" / "你怎么看，欢迎在评论区留言"

## 5. 最终 JSON 输出契约

最终输出必须只包含一个合法的 JSON 对象，不带有任何 Markdown 代码块包裹，也不含任何前置或后置说明。

```json
{
  "source_name": "报告本身标题（不带后缀，如：腾讯2025年第四季度财报）",
  "video_titles": ["3条发布标题候选，突出行业异动或利益关联，互不重复"],
  "cover_titles": ["3条封面主标题，4到10字，强冲突感"],
  "cover_subtitles": ["3条封面副标题，15字以内，用核心数据背书"],
  "golden_quotes": ["3条解说金句，直击商业本质或经济因果的真话"],
  "comment_hook_options": ["3条评论引导，抛出利益分歧、行业争议或民生对照，促使观众争论"],
  "share_hook_options": ["3条转发引导，提供社交货币，让人想发给同行、老板或家人"],
  "content": "完整口播终稿，按 final_target_chars 左右控制字数，不分段，无 Markdown，所有数字与数据对照表完全吻合",
  "total_length": 0,
  "target_segments": 0
}
```

### JSON 规范细则
- `content` 去掉换行符与多余空白，是一个连续的适合口播的字符串。
- `total_length` 记录 `content` 的真实字符数。
- 候选数组中的每一项（如标题、金句）必须各提供 3 条，且互不重复。
- 确保 JSON 绝对可被 Python `json.loads()` 解析，不可出现多余逗号。
